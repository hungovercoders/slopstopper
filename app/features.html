🔒 Security

Block vulnerable code and leaked credentials before they merge — static analysis, dynamic scanning, secrets detection and dependency CVE checks on every change.

  • SAST — Semgrep flags vulnerabilities in source code before merge
  • DAST — OWASP ZAP probes a running site for common web vulnerabilities
  • Secrets — Gitleaks scans every commit for accidentally committed credentials
  • Dependencies — Trivy checks all dependencies for known CVEs
Learn more about Security →
SAST (Semgrep) 
name: SlopStopper · SAST Analysis
on:
  pull_request: { branches: [ main ] }
  push:         { branches: [ main ] }
jobs:
  sast:
    name: Static Application Security Testing with Semgrep
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install "https://github.com/hungovercoders/slopstopper/releases/download/v0.2.1/slopstopper_cli-0.2.1-py3-none-any.whl" && pip install semgrep
      - run: slopstopper run security:sast
      - if: github.event_name == 'pull_request'
        run: slopstopper emit security:sast --target pr-comment

🧹 Hygiene

Keep the codebase auditable and the docs honest — complexity caps, structure / accuracy / size checks for docs, and auto-labelled PRs.

  • Complexity — Lizard + Bandit cap cyclomatic complexity and Python security smells
  • Doc Structure — Enforces docs/index.md as the documentation map; README / AGENTS.md / CLAUDE.md stay thin pointers
  • Doc Accuracy — Documentation reflects the actual code
  • Doc Size — Prevents documentation from going stale or unwieldy
  • CSP Exceptions Drift — Keeps worker/headers.json and CSP_EXCEPTIONS.md in sync so per-page CSP relaxations stay documented
  • Auto-label PRs — PRs labelled from the paths changed

CSP Exceptions Drift 
name: SlopStopper · Hygiene - CSP Exceptions Drift Check
on:
  pull_request:
    paths:
      - 'worker/headers.json'
      - 'docs/security/CSP_EXCEPTIONS.md'
jobs:
  check-csp-exceptions:
    steps:
      - uses: actions/checkout@v4
      - run: pip install "https://github.com/hungovercoders/slopstopper/releases/download/v0.2.1/slopstopper_cli-0.2.1-py3-none-any.whl"
      - run: slopstopper run hygiene:csp-exceptions
      - if: github.event_name == 'pull_request'
        run: slopstopper emit hygiene:csp-exceptions --target pr-comment

Strict default-src 'self' CSP is the default. Per-path exceptions (e.g. Giscus on /feedback.html) must be documented in CSP_EXCEPTIONS.md with SRI hashes and a fallback. This check fails the build if worker/headers.json and CSP_EXCEPTIONS.md disagree.

✅ Reliability

Catch broken pages, dead internal links, slow loads, accessibility regressions and broken social-share previews before users do — E2E, smoke, broken-link, axe-core WCAG 2.1 AA, Lighthouse CI and SEO/OpenGraph checks on every change.

  • E2E Tests — Playwright covers navigation and critical user journeys
  • Smoke Tests — Lightweight checks after every deploy
  • Broken-Link Checks — Crawls internal anchors from key pages and fails on 4xx/5xx destinations
  • Accessibility — axe-core audits every page against WCAG 2.1 AA
  • Core Web Vitals — Lighthouse CI measures LCP, INP and CLS
  • SEO & Social-Share — Lighthouse SEO budget + OpenGraph / Twitter Card / canonical presence check

🚀 Deployment

Every push deploys, every PR gets a preview URL, every PR-close cleans up — Cloudflare Workers Builds, hooked directly into the repo. No deploy workflow, no secrets.

  • Production Deploy — Merges to main trigger an automatic release
  • Preview Deploys — Every PR gets a unique preview URL as a commit check
  • Auto Cleanup — Preview deploys are removed automatically on PR close
Learn more about Deployment →
wrangler.jsonc 
{
  "name": "slopstopper",
  "main": "worker/index.ts",
  "compatibility_date": "2026-06-01",
  "assets": {
    "directory": "./app",
    "binding": "ASSETS",
    "run_worker_first": true
  }
}

Cloudflare Workers Builds reads this file from the repo on every push, runs npm run build, and deploys the Worker with the app/ directory bound as static assets. No GitHub Action involved.

Development loops

Inner loop. Outer loop. Both fast.

SlopStopper organises quality feedback into two loops that keep velocity high and quality consistent.

Inner Loop — Local

The fast, local cycle a developer (or AI agent) runs before pushing. Completes in seconds to minutes.

🔨 Build & Lintlocally
🧪 Run Testslocally
📤 Commit & Pushopen PR
↩ iterate on feedback

Outer Loop — CI/CD

Automated checks triggered on every push or PR. Deterministic feedback before code reaches production.

✅ ReliabilityE2E · Smoke · A11y · CWV

One source of truth

Same commands. Both loops.

Every check is a task ss:<name> command — defined once in Taskfile.ss.yml under the ss namespace. You type it locally for instant feedback. CI runs the exact same command. No drift between what you ran and what the pipeline runs — fast inner loop, consistent outer loop, identical behaviour.

🧑‍💻 Inner loop — local
  • task ss:hygiene:complexity
  • task ss:security:sast
  • task ss:reliability:accessibility
  • task ss:reliability:smoke
Taskfile.ss.yml one definition
🤖 Outer loop — CI
  • task ss:hygiene:complexity
  • task ss:security:sast
  • task ss:reliability:accessibility:ci
  • task ss:reliability:smoke:ci
What a Taskfile entry looks like

Definitions inside Taskfile.ss.yml use bare names. The root Taskfile.yml imports them under the ss namespace via includes:, so you invoke them with the prefix (task ss:reliability:accessibility):

# Taskfile.ss.yml — bare names, defined under the `ss` namespace
reliability:accessibility:
  desc: Run axe-core accessibility audit against a URL
  cmds:
    - npx playwright test tests/accessibility.spec.ts

reliability:accessibility:ci:
  desc: Same audit, wired for CI artifacts
  cmds:
    - task: reliability:accessibility

And the wiring in the root Taskfile.yml is just:

# Taskfile.yml — thin integration layer
version: '3'
includes:
  ss:
    taskfile: ./Taskfile.ss.yml
